# Activity Application Skill

## 目标 (Goal)
协助用户高效完成各类活动及奖项的申请工作。通过分析活动规则和用户提供的案例资料，生成符合要求、高质量的申请文档。

## 核心工作流 (Core Workflow)

### 第一步：接收与分析活动规则 (Rule Analysis)
当用户提供活动规则时，根据输入类型采取不同策略：


1.  **文本输入**：
    - 直接分析用户提供的文本描述。
2.  **文件与图片输入**：
    - **文档 (Word/PDF)**：尝试读取文件内容。如果是扫描件或不可选中的文本，尝试进行识别。
    - **填空模板 (Template)**：如果用户提供了活动方官方 Word 模板（如“申请表.docx”），则跳过结构分析，直接识别模板中的“填空项”和“说明文字”，准备进行定点填空。
    - **图片 (JPG/PNG/Screenshot)**：活动规则常以海报或长图形式存在。尝试分析图片中的文字信息。
    - **注意**：如果无法解析文件内容，请礼貌请求用户提供文本内容或关键信息摘要。
3.  **网站链接 (URL)**：
    - 使用 `WebFetch` 或 `cursor-ide-browser` 工具访问链接，抓取活动页面内容。
    - 重点关注：“参评条件”、“评分标准”、“材料要求”、“提交截止时间”。

规则来源可能为文字、文档、URL 或图片海报，请优先按规则文件处理，不要混同为案例资料。

**分析输出要求**：
- **关键时间点**：报名截止、资料提交截止。
- **核心门槛**：必须满足的硬性条件。
- **评分维度**：评审关注的重点（如：创新性占 40%，应用效果占 30%）。
- **材料清单**：明确需要提交哪些文档、模块内容、证明材料。
- **材料信息要求**：明确所需材料格式、字数、字号等信息。


### 第二步：接收与分析活动资料 (Case Analysis)
接收用户提供的活动资料（支持图片、网站、PPT、Word、Excel 等格式的路径或描述）。

1.  **资料读取**：
    - **Office 文档**：对于 Word/Excel/PPT，优先使用 `read_file`。如果内容乱码或无法读取，转而使用 `scripts/parse_office.py` 脚本提取纯文本。
    - **PDF 文档**：直接使用 `read_file` 读取。
    - **图片**：直接使用 `read_file` 读取，利用视觉能力分析图表数据和文字。
    - **网站**：使用 `WebFetch` 抓取。
2.  **资料匹配**：
    - 将活动资料与**第一步**中的“材料清单”进行初步圈定和认定。
    - 将案例资料与**第一步**中的“评分维度”进行对齐。
    - 找出案例中的“得分点”和“薄弱点”。
    - 如果资料缺失（例如规则要求提供“用户反馈”、”公司介绍“等信息，但案例资料中没有），主动提示用户补充。

### 第三步：撰写申请资料 (Drafting Application)
根据规则要求和案例素材，生成申请文档草稿。

**撰写原则**：
1.  **结构清晰**：严格按照活动方要求的模板或章节结构书写。如果没有模板，采用“背景-目标-方案-创新点-成效-总结”的标准结构。
2.  **客观量化**：多用数据证明效果（如“效率提升 50%”，“节省成本 100 万”）。
3.  **突出亮点**：针对评分权重高的部分，加大篇幅和深度。
4.  **语言风格**：专业、简洁、有说服力。
5.  **严格合规检查 (Compliance Check)**：
    - **字数控制**：撰写内容时需实时预估篇幅，确保核心章节字数达到规则要求的 90% 以上。
    - **排版提示**：由于输出格式限制，**必须**在文档末尾附带【排版与提交指南】章节，明确列出活动方对“资料格式”、“字号”、“字体”、“颜色”等具体要求，提醒用户在最终排版时执行。
6.  **模板填充 (Template Filling)**：
    - **核心逻辑**：**绝不**重新创建文档。必须基于用户提供的 `.docx` 文件进行修改。
    - **防呆机制**：如果无法确定替换位置，或替换后格式错乱，请直接**复制**原模板作为输出，并生成一份单独的文本文件说明填空内容。**严禁**因为处理不了模板而退化为生成无格式的纯文本 Word。

**输出格式 (Output Rules)**：
**必须** 默认生成 `.docx` 文件。
- **场景 A：用户提供了模板（必须保留格式）**
  **优先级：最高 (High)**
  `python3 scripts/create_docx.py 填好的申请书.docx --template 用户提供的模板.docx --replace "原标题::新标题" --replace "项目背景::项目背景：...(新内容)..."`
  *(调试提示：如果替换未生效，可能是因为 Word 将占位符拆分成了多个 XML 标签。此时请保留原模板输出，告知用户手动填写。)*

- **场景 B：用户未提供模板（新建文档）**
  **优先级：低 (Low)**
  当用户没有提供模版或者没有说明需要输出的内容时，给一个默认的模版，并告知用户。
  `python3 scripts/create_docx.py 申请资料_草稿.docx "标题" "段落内容..."`

## 格式兼容性 (Format Compatibility)

为确保文档的稳定性和可读性，本 Skill 采用以下标准：

1.  **输入兼容性**：
    - **支持**：Office Open XML 格式 (`.docx`, `.xlsx`, `.pptx`)、PDF、纯文本、图片。
    - **不支持**：旧版二进制格式 (`.doc`, `.xls`, `.ppt`)。
    - **对策**：当检测到二进制格式输入时，**Agent 必须** 明确告知用户：“无法直接处理旧版二进制格式，请将文件另存为 .docx 或导出为文本/PDF 后重新上传。”

2.  **输出兼容性**：
    - 始终生成标准的 `.docx` 文件。
    - 如果用户需要 `.doc` 格式（极少见），请建议用户在收到 `.docx` 后自行使用 Word 软件进行“另存为”操作。

## 可用脚本与工具 (Tools & Utilities)
当文档无法直接解析时，根据文件类型选择合适的工具：

1. **Office 文档 (.docx, .xlsx, .pptx)**
   使用内置通用脚本进行解析：
   `python3 scripts/parse_office.py /path/to/file.ext > out.txt`
   *注意：如果遇到二进制旧格式 (.doc, .xls, .ppt)，脚本会报错。请引导用户另存为新版 XML 格式或导出 PDF。*

2. **PDF 文档**
   直接使用 `read_file` 工具读取，系统会自动提取文本。

3. **图片 (JPG, PNG, Screenshot)**
   直接使用 `read_file` 工具读取图片路径。Agent 具备视觉能力，可直接从图片中提取文字和图表信息。

4. **网站链接 (URL)**
   使用 `WebFetch` 工具抓取内容。

## 交互示例 (Interaction Example)

**用户**：我要申请“2026年度最佳创新项目奖”，这是活动官网链接 [url] 和我的项目资料 [file_path]。

**Agent**：
1.  **动作**：调用工具读取官网规则和本地项目资料。
2.  **回复**：
    > 已获取活动规则。该奖项重点关注“技术突破性”（40分）和“商业价值”（30分）。
    >
    > 您的项目资料中，“AI算法优化”部分非常契合技术突破性，但在“商业变现”方面的数据略显单薄。
    >
    > **建议补充**：目前的营收数据或用户增长趋势图。
    >
    > 是否现在开始为您撰写申请草稿？

## 提示与注意事项 (Tips)
- 在处理多个文件时，保持耐心，逐一确认文件内容。
- 遇到无法识别的文件格式，引导用户提供纯文本或转换后的 PDF。
- 始终以“帮助用户获奖”为最终目标，不仅仅是填空，而是进行优化和润色。
