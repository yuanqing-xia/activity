# Activity Application Skill

## Goal

协助用户高质量完成活动、奖项、荣誉、案例征集等申请工作。核心任务是：读懂活动规则，优先从产品知识库获取我方产品事实，再结合用户提供的案例资料和证明材料，生成符合格式、突出评分点、可追溯的申请文档。

## Source Policy

涉及“我方产品资料、产品能力、解决方案、功能亮点、适用场景、产品价值、技术架构、客户案例、指标口径”等内容时，优先查询产品知识库：

- GitHub repository: `qschouteam/Aicare-Product-Kb`
- URL: `https://github.com/qschouteam/Aicare-Product-Kb`
- Default branch: `master`
- Repository visibility may be private; use the user's authenticated GitHub account/session when needed.

资料优先级：

1. 活动方官方规则、模板、通知。
2. `qschouteam/Aicare-Product-Kb` 中的产品事实。
3. 用户提供的案例资料、证明材料、项目材料。
4. Agent 基于材料的归纳与表达。

不要凭记忆编造产品能力、指标、客户、奖项、资质、合作方或效果数据。知识库没有证据时，必须标注为“待补充”或向用户索要证明材料。

## Core Workflow

### 1. Analyze Activity Rules

当用户提供活动规则时，先判断来源类型：

- 文本：直接分析。
- Word/PDF：读取正文；如为扫描件或不可选文本，尝试识别或请用户补充文字版。
- 官方 Word 模板：识别填空项、说明文字、格式要求和限制条件；后续必须基于原模板填充。
- 图片/海报/截图：提取文字，重点识别时间、条件、材料、评分标准。
- URL：访问活动页面，抓取“参评条件、评分标准、材料要求、提交截止时间、格式要求”。

规则分析输出必须包含：

- 关键时间点：报名截止、材料提交截止、答辩/公示时间。
- 硬性门槛：主体资格、产品/案例范围、时间范围、地区/行业限制。
- 评分维度：权重、评审关注点、加分项。
- 材料清单：申请表、案例正文、证明材料、营业/资质材料、图片/视频等。
- 格式要求：字数、字体、字号、页数、文件格式、命名规则、盖章/签字要求。
- 风险项：不满足条件、缺材料、超字数、需盖章、需敏感信息脱敏等。

### 2. Query Product Knowledge Base

当申请内容需要我方产品信息时，先检索 `qschouteam/Aicare-Product-Kb`，再写作。该知识库可能是私有仓库：如果 GitHub connector、未登录浏览器、`curl` 或裸 API 返回 404，不要直接判定仓库不存在；改用用户已登录的 GitHub 浏览器会话或本地安装版。

可用访问通道按优先级选择：

1. 本地安装版：`~/.agent-skills/aicare-product-kb/`、`~/.codex/skills/aicare-product-kb/` 或用户提供的 clone 路径。
2. 用户已登录的 GitHub 页面：`https://github.com/qschouteam/Aicare-Product-Kb`。
3. 已授权的 GitHub connector/API。
4. 用户导出的知识库文件或粘贴内容。

如果发现本地安装版，优先用 `rg` 检索，并在可行时运行知识库自带 freshness 检查：

```bash
cd ~/.agent-skills/aicare-product-kb && ./check-fresh.sh
rg -n "关键词1|关键词2" ~/.agent-skills/aicare-product-kb
```

推荐检索顺序：

1. 根据活动主题和评分维度提炼关键词，例如：产品名称、场景、行业、核心功能、技术能力、合规、安全、效果指标。
2. 先看知识库 `README.md`、`SKILL.md` 和 `references/` 目录索引，再查产品介绍、方案文档、FAQ、案例、指标说明、版本说明等。
3. 对同一关键事实尽量找到最直接的来源文件；必要时交叉查看多个文件。
4. 记录可引用事实，包括：事实内容、来源文件/链接、适合用于哪个评分维度、是否还需要用户提供证明。

如果当前环境无法访问知识库：

- 先说明“当前通道无法访问”，并具体指出是 connector、CLI、浏览器还是本地路径不可用。
- 优先询问用户是否已在 in-app browser 登录 GitHub，或是否已安装知识库到 `~/.agent-skills/aicare-product-kb/`。
- 请用户提供访问权限、本地 clone 路径、导出的知识库文件，或粘贴相关产品资料。
- 在未获得来源前，不要把产品事实写成确定表述。

检索后形成内部事实表：

- `事实`：可直接写入申请书的产品信息。
- `来源`：知识库文件路径、链接或用户文件名。
- `用途`：对应活动评分维度或材料章节。
- `证据状态`：已由知识库支持 / 需要案例材料证明 / 待用户补充。

如当前项目仓库可写，建议将重要检索摘要保存到 `materials/<活动名>/product-kb-notes.md`，便于审计和复用。

### 3. Analyze Case Materials

接收用户提供的案例资料、证明材料、截图、官网、PPT、Word、Excel、PDF、图片等。

资料读取策略：

- Office Open XML (`.docx`, `.xlsx`, `.pptx`)：优先直接读取；如读取失败，使用 `scripts/parse_office.py` 提取文本。
- PDF：直接读取文本；扫描件需识别或请用户提供文字版。
- 图片：读取并提取文字、图表、界面信息。
- URL：抓取页面内容，保留关键链接。
- 旧版二进制 Office (`.doc`, `.xls`, `.ppt`)：告知用户另存为 `.docx/.xlsx/.pptx` 或导出 PDF/文本。

分析重点：

- 将案例资料与活动材料清单逐项匹配。
- 将产品知识库事实与案例证据对齐，区分“产品能力”与“项目成效”。
- 找出得分点：创新性、先进性、应用规模、商业价值、社会价值、合规安全、可复制性等。
- 找出薄弱点：缺少量化指标、缺少截图/合同/客户反馈、缺少时间范围、缺少盖章证明等。
- 主动提示用户补充缺失材料，不要用虚构内容填补。

### 4. Draft Application

根据活动规则、产品知识库事实和案例材料撰写申请草稿。

写作原则：

- 严格按活动方模板或章节结构写；无模板时使用“背景-痛点-方案-产品能力-创新点-应用成效-可复制性-总结”。
- 产品能力来自知识库，项目成效来自案例/证明材料；两者不要混写成无来源结论。
- 针对高权重评分维度加大篇幅和论证密度。
- 尽量量化，但只使用有来源的数据；无来源数据标为待补充。
- 语言专业、简洁、有说服力，避免营销空话。
- 字数控制达到规则要求的 90% 以上，且不超过上限。
- 文档末尾必须附【排版与提交指南】，列出活动方格式、字号、字体、页数、命名、盖章、附件要求。
- 如存在缺口，附【待补充材料清单】。

### 5. Fill Word Templates

如果用户提供官方 `.docx` 模板，优先级最高：必须基于原模板修改，不要重新创建文档。

推荐命令：

```bash
python3 scripts/create_docx.py 填好的申请书.docx --template 用户提供的模板.docx --replace "原标题::新标题" --replace "项目背景::项目背景：...(新内容)..."
```

如果替换位置不确定、占位符被 Word 拆分、或替换后格式错乱：

- 复制原模板作为输出。
- 另生成一份文本说明，列出每个填空项建议填写内容。
- 不要退化为无格式纯文本 Word。

没有模板时才新建 `.docx`：

```bash
python3 scripts/create_docx.py 申请资料_草稿.docx "标题" "段落内容..."
```

## Output Requirements

默认输出 `.docx` 文件，并在回复中简要说明：

- 已使用的活动规则来源。
- 已使用的产品知识库来源或无法访问说明。
- 已使用的用户案例资料。
- 仍需用户补充的材料。
- 生成文件路径。

如内容中包含来自知识库的关键产品事实，在草稿或说明中保留可追溯来源，例如“来源：`qschouteam/Aicare-Product-Kb/<path>`”。

## Scripts

- `scripts/parse_office.py /path/to/file.ext > out.txt`：提取 `.docx/.xlsx/.pptx` 纯文本。
- `scripts/create_docx.py`：新建 `.docx` 或基于模板填充 `.docx`。

## Interaction Example

用户：我要申请某创新案例奖，这是活动规则链接和官方模板。产品资料从我们的知识库查，案例效果数据我稍后补。

Agent：

1. 读取活动规则和模板，提取评分维度、材料清单、格式要求。
2. 检索 `qschouteam/Aicare-Product-Kb`，整理与“创新性、产品能力、应用场景”相关的产品事实。
3. 标注缺口：效果数据、客户反馈、上线时间、截图证明待补充。
4. 基于原模板生成 `.docx` 草稿，并附【排版与提交指南】和【待补充材料清单】。

## Guardrails

- 不要把活动规则文件误当作案例资料。
- 不要把产品知识库中的通用能力写成特定客户成效，除非案例材料证明。
- 不要编造客户名称、指标、合作方、资质、奖项、认证。
- 不要忽略官方模板格式。
- 不要在无法访问知识库时假装已检索。
- 始终以提高获奖/入选概率为目标：不仅填空，也要发现缺口、强化论证、提示补证。
